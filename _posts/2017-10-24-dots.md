Dotfiles! They're fantastic, they let you configure your system to be _just_ the way
you like it. Which I love. The downside is that once you are used to your system
running just the way you like it, you tend to be a bit lost without all of your
config... So, what can we do to always have our config with us?

_Clearly_ the answer is to have a dotfile for our dotfiles.


# Yo dog, let's get started
If you want to follow along, the shell script that makes all of this work (TM,
patent pending, Kickstarter launching soon etc...) can be found [here][0]. The
idea is to manage a git repo that contains our dotfiles in a quick and easy way
that makes it simple to see what we currently have under source control.

With the `dots` script mentioned earlier on your `$PATH` let's initialise the repo:

```bash
$ dots init
Initialising ~/.dots repo...
Initialised empty Git repository in /home/innes/.dots/
Creating ~/.my-dots...
Done

$ cat .my-dots
.my-dots
```

The default locations of `~/.dots` and `~/.my-dots` are configured in the script
if you would prefer a different name but both _must_ be in your `$HOME`
directory for this to work.

Now that we have our `.my-dots` dotfile-dotfile, lets add in some files to
track:

```bash
# Contents of .my-dots
.my-dots
.gitconfig
.vimrc
.zshrc
bin/
```

And commit those into the repo:

```bash
$ dots update
Updating dotfiles found in ~/.my-dots...
[master a91c710] Updating dots: Thu Oct 26 09:10:23 BST 2017
 30 files changed, 3521 insertions(+)
 create mode 100644 .gitconfig
 create mode 100644 .vimrc
 create mode 100644 .zshrc
 create mode 100755 bin/anim_plot
 create mode 100755 bin/dots
 create mode 100755 bin/gimme
 create mode 100755 bin/issue
 create mode 100755 bin/note
 create mode 100755 bin/spark
 create mode 100755 bin/terraform
 create mode 100755 bin/yoump3

 Done
```

If all you want is local version control of your dots then this is it! Under the
hood, `dots` is a wrapper around `git` that allows you to add any file under your
home directory that you list in the `.my-dots` file. Other than three special
cases (`init`, `update` and `remote` which we'll see later) all other arguments are
passed straight through to `git`:

```bash
$ dots --no-pager log --stat
```

You can checkout, revert and branch to your heart's content! If you want to be
able to access your dotfiles on another machine however, then keep reading...


# Setting the remote
Head to your cloud git provider of choice (I've used [github][1] here though I keep
my personal dotfiles in [bitbucket][2] as I have several sensitive aliases and
environment variables that I need to keep private) and create a new repository:

![github-repo](/assets/dots_files/github-repo.png)

Once you've created the repository, you want to look for the link provided for
pushing an existing repo from the command line. It should look something like this:

![github-repo](/assets/dots_files/github-push.png)

With the link in hand, we'll add the remote to our dotfile repo using `dots remote`:

```bash
$ dots remote git@github.com:sminez/my-dotfiles.git
Adding remote repo: git@github.com:sminez/my-dotfiles.git ...
Making initial commit...
Counting objects: 40, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (36/36), done.
Writing objects: 100% (40/40), 12.88 MiB | 2.88 MiB/s, done.
Total 40 (delta 4), reused 0 (delta 0)
remote: Resolving deltas: 100% (4/4), done.
To git@github.com:sminez/my-dotfiles.git
 * [new branch]      master -> master
Branch master set up to track remote branch master from origin.
Done
```

And we're done! Now that we have a remote added to the dots repo every time that
we run `dots update` we'll also update the remote repo as well.

![github-repo](/assets/dots_files/github-final.png)
